## What Went Well

- **Investigation-first scope.** The epic was born from a real bug plus an investigation doc that pre-localized every root cause (model/DB schema drift, the IntegrityError masquerade, the missing project_admin linkage). `dev-story` never re-investigated ‚Äî it executed. Five surgical stories, no scope creep.
- **Reuse over reinvention (again).** 15.5 mirrored the project-card edit/delete pattern; 15.3 + 15.5 reused the `ProjectMembership(role="project_admin")` linkage and the `PROJECT_ADMIN_ROLE` constant from the signed-off RBAC design ‚Äî no new table, and the only migration was the one bugfix.
- **Clean single-file coordination.** Four stories (15.2/15.3/15.4/15.5) edited `AdminDashboard.tsx` without clobbering each other, because each story's notes carried explicit coordination (sequence 15.1‚Üí15.5, plus "keep `isAdminUser` ‚Äî 15.5 needs it"). Cheap, high-leverage discipline.
- **Hardening beyond the spec.** 15.5 caught the `uq_project_memberships_project_user` UNIQUE hazard and switched standard‚Üíproject_admin promotion to promote-in-place (avoiding a 500 on an existing (project,user) row). 15.3's review caught `db.flush()` sitting outside the `try/except IntegrityError` (an email race would 500 instead of 409).
- **"Run the whole suite" paid off.** The full regression at 15.5 caught a cross-story contract regression ‚Äî a pre-15.3 RBAC test that created a project_admin without a project ‚Äî and fixed it. A subset run would have missed it.
- **Security and full-stack sync held.** Secret-free `_to_admin_user_response`, no password echo, immutable platform admin, self-lockout guards; every backend payload change updated the matching TS interface in the same change.

## Challenges & Growth Areas

- **Schema-drift is invisible to the test suite (top lesson).** The actual production bug in 15.1 ‚Äî `projects.confluence_base_url` still `NOT NULL` on PostgreSQL ‚Äî could not be reproduced or confirmed by pytest, because SQLite builds the schema from the ORM model (already `nullable=True`). The fix relies on Thuong's live PostgreSQL migrate-and-create for confirmation. This is the Epic 14 "hermetic-green ‚âÝ integration-validated" lesson in a new form: model/DB drift, not just integration behavior.
- **Live validation still pending across the board.** The admin-dashboard changes are unit/component-green but not live-validated on Win11; the migration is unapplied. Combined with Epic 14's still-outstanding live Jack run, integration confidence keeps deferring.
- **Compounding release debt.** Epic 14 retro action item #1 (commit + `alembic upgrade head`) was not done. There are now **3 migrations pending**, all uncommitted ‚Äî the code‚ÜîDB gap widened by another epic.
- **Review depth under-weighted the riskiest story.** Epic 15 used lighter per-story inline review (vs Epic 14's adversarial sweep). The highest-risk story, 15.5 (role-flip, lockout, immutability), left 10 deferred items. Most are genuinely fine (defense-in-depth dead code, admin-only N+1), but the volume signals the riskiest surface got the least adversarial scrutiny relative to its size.
- **Renumber-induced staleness (meta, repeated within 24h).** The Epic 14 retro (same day, 2026-06-21) predicted "Epic 15 = Conversational UX" and wrote story-creation guidance for it; a same-day renumber inserted *this* Epic 15 (Admin RBAC) and pushed Conversational UX to Epic 16. So Epic 14 retro action item #3 is mis-numbered ‚Äî it applies to Epic 16. The Epic 14 retro had itself listed "prior-retro previews go stale after reprioritization" as a key insight, and it recurred immediately.

## Key Insights

1. **Schema drift is a distinct blind spot from logic bugs.** The SQLite test harness cannot see PostgreSQL `NOT NULL`/constraint drift. Guard it with metadata assertions (`Column.nullable is True`) and treat the live migrate-and-create as a first-class deliverable ‚Äî generalizes Epic 14's hermetic-green lesson.
2. **A pre-localized investigation turns dev-story into pure execution.** Front-loading root-cause analysis before decomposition removed re-investigation entirely and kept stories surgical.
3. **Sibling stories editing one file are safe when the story notes carry explicit coordination** (sequence + "don't delete X"). Low cost, prevents clobbering.
4. **Scale review depth to story risk, not uniformly.** Per-story inline review under-weights the riskiest story. The single highest-risk story in an epic warrants an adversarial pass.
5. **Renumbering invalidates cross-document forward references by number.** Re-read a prior retro's next-epic guidance against current `epics.md` numbering ‚Äî never trust the written number after a reprioritization.
