## Story

As a QA user,
I want to review each generated Playwright script **beside its source test case** — with the Python script **syntax-highlighted**, **Next/Previous** navigation between script/test-case pairs, a **per-item review status**, and any review **warnings/TODO markers shown without hiding the script** —
so that I can verify traceability and correctness before approving.

## Acceptance Criteria

Verbatim from [epics.md#Story-13.5](_bmad-output/planning-artifacts/epics.md) (lines 1345-1365), expanded with implementation defaults (see "Scope decisions" — **all four defaults CONFIRMED by Thuong 2026-06-13** ("dùng toàn bộ default"); no pending input remains, see "Confirmed decisions" at the end of this file). This is the **capstone full-stack review-UX story for Sarah** — the script-review analog of **Story 12.4** (Mary's `MaryReviewPanel`) and of **Story 11.6** (Bob's `SplitPanel`). It builds the rich side-by-side **script-review panel** + its strongly-typed TS interfaces + the present-all review transport, on top of the Sarah step-4 frontend surface that **Story 13.1 creates** and the `warnings` channel that **Story 13.2** populates.

### AC1 — Side-by-side: source test case (left) ↔ syntax-highlighted Python script (right)

- **Given** Sarah generated one or more scripts
- **When** the review UI opens
- **Then** the **source test case** is shown on one side (left) and the **generated Python script** on the other (right) — the structured test case (title, objective, preconditions, steps, expected results, source-requirement reference) on the left, the generated script on the right
- **And** the generated script content has **Python syntax highlighting** (using the already-installed `react-syntax-highlighter`; **no new package** — see Dev Notes "Syntax highlighting")

### AC2 — Multi-script navigation + per-item review status

- **Given** multiple generated scripts exist
- **When** the user reviews them
- **Then** the user can navigate between script/test-case pairs with **Next/Previous** controls (client-owned, mirroring `SplitPanel`/`MaryReviewPanel`)
- **And** **review status is visible for each item** (e.g. approved / skipped / pending), not just for the currently-shown script — which requires the client to hold the **full set** of scripts (the present-all transport; see "The key design decision")

### AC3 — Warnings/TODO markers visible without hiding the script

- **Given** the generated script contains warnings or `# TODO:` / `# REVIEW:` markers (the `GeneratedScript.warnings` channel established by **13.2** and specialized by **13.3** selectors/assertions / **13.4** SSO-credentials)
- **When** the script is reviewed
- **Then** the warnings are **visible** (an advisory amber banner tied to the script, plus the inline `# TODO:`/`# REVIEW:` markers preserved in the highlighted script body) **without hiding the script content** — the warning surface never replaces or collapses the script pane

## Scope decisions (CONFIRMED — Thuong locked all four defaults 2026-06-13)

Chosen from the code + ACs + planning docs + the 12.4/11.6 sibling precedent, and **confirmed by Thuong** ("dùng toàn bộ default", 2026-06-13). The four formerly-open questions are now resolved decisions (full list under "Confirmed decisions" at the end of this file). No pending input — the dev agent implements exactly as written.

- **This is the capstone full-stack review-UX story for Sarah (mirror 12.4 / 11.6).** The work is: (a) **build the rich side-by-side script-review panel** `frontend/src/components/agents/SarahScriptReviewPanel.tsx` (sibling to 13.1's `SarahInputSelection.tsx`); (b) add its strongly-typed TS interfaces; (c) render the generated Python with **syntax highlighting** via the already-installed `react-syntax-highlighter` (no new package); (d) provide **client-owned Next/Previous** + a **per-item review-status** indicator; (e) surface the **`warnings` channel** (13.2/13.3/13.4) as an advisory amber banner without hiding the script; and (f) two focused **backend** changes in `sarah.py`: the **present-all review transport** (`_present_script_review`, analogous to Mary's `test_case_review` and Bob's `is_review_ready`+`pages`) and **index/id-addressable** approve/reject/skip (so a freely-navigated card maps to the right script), keyed off a `_reviewed_indices` set for the DONE gate. The **Approve/Reject/Skip action footer is wired to the existing Epic-5 handlers** (made index-addressable) so the panel is functional end-to-end — but the *refinement* of those flows is fenced to later stories (see below). (Decision #1, #2.)
- **Review transport = present-all + client-side navigation + index-addressable approve/reject/skip (recommended default, mirrors Bob/Mary/12.4).** The live Sarah backend presents scripts **one at a time** (`_present_current_script_for_review` emits `metadata.type=="review_request"` + a single-script `review_data`; `handle_navigate` does backend Prev/Next round-trips). 13.5 switches Sarah to the **present-all** transport the rest of the project uses: emit the **full** `self._generated_scripts` list in one `metadata.type=="script_review"` payload; the FE panel owns Prev/Next + the resolved set. This is **required** to satisfy AC2's "review status visible for **each** item" (the client must hold the whole set) and is consistent with the project's deliberate Bob/Mary pattern. Because the user can navigate freely, approve/reject/skip carry `data["script_index"]` (default = `_current_review_index` for back-compat), paired with a new `_reviewed_indices: set[int]` that becomes authoritative for the DONE gate (mirrors 12.4's `_reviewed_indices`). See "The key design decision" + Saved Question #1.
- **Boundary fences — what 13.5 does NOT change.** Story 13.1's note designates the per-item **script** review loop as "Epic-5 + **13.5+/13.7** territory"; 13.5 owns the **transport + addressing + rendering**, and explicitly **defers**: the **edit-before-approval** affordance (Story 13.6 — no editable script pane here), the **refined approve/reject/regenerate semantics** — approval metadata (user + timestamp), feedback-into-the-generation-prompt, "rejected output is never treated as approved", and the Jack-eligibility filter (**Story 13.7**), the **artifact-save metadata** + the `.spec.ts` save-fallback defect ([sarah.py:538](src/ai_qa/agents/sarah.py:538), **Story 13.8**), the **confidence engine** (`_calculate_confidence` — Epic 5, fenced by 13.2/13.3/13.4; 13.5 only *renders* the float), and the **warnings detectors/prompt** (13.2/13.3/13.4 — 13.5 only *renders* `warnings`). The existing Epic-5 approve/reject/skip **behavior** (save on approve, regenerate on reject, advance, DONE) is **preserved** — 13.5 makes it index-addressable and switches the transport, nothing more.
- **Syntax highlighting needs NO new package (project rule: no new packages).** `react-syntax-highlighter` (`^16.1.1`) + `@types/react-syntax-highlighter` are already installed and already used to render raw `.py` content in [ArtifactPreview.tsx:194-206](frontend/src/components/artifacts/ArtifactPreview.tsx:194) (`<SyntaxHighlighter language="python" style={vscDarkPlus} PreTag="div" wrapLongLines>`). 13.5 mirrors that exact pattern. Do **not** add `prismjs`/`shiki`/`monaco`/`codemirror`. See Dev Notes "Syntax highlighting".
- **Per-item review status + confidence are rendered client-side from already-emitted data (no engine change).** Review status (approved / skipped / pending) is derived from each script's `approved` flag + the `_reviewed_indices` set carried in the present-all payload. The **green/amber/red confidence badge** maps the already-emitted per-script `confidence` **float** ([sarah.py:724](src/ai_qa/agents/sarah.py:724)) to a level **client-side** (there is no script `confidence_level` and 13.5 does **not** add one — that would touch the confidence engine the prior stories fenced). Note: confidence display is **not** a 13.5 AC; it is a natural addition using existing data — Saved Question #3.
- **Out of scope:** Sarah's input-selection panel (13.1 owns `SarahInputSelection.tsx`), the Chrome-path **frontend** UI (13.1 confirmed deferred — a generation prerequisite; the script review is only reachable when a Chrome path is already configured so the chrome-path request is skipped — see Dev Notes "Reaching the review"), the generation prompt/selectors/SSO (13.2/13.3/13.4), edit (13.6), refined approve/reject/regenerate semantics + approval metadata + Jack-eligibility (13.7), artifact-save metadata (13.8). **No Alembic migration** (no model/DB change; the review payload rides the existing WebSocket metadata channel).

### The key design decision — review transport (most load-bearing)

The live Sarah backend presents scripts **one at a time** (`_present_current_script_for_review` + backend `handle_navigate`). A rich, navigable side-by-side card where **per-item review status is visible** (AC2) needs the **whole set** on the client. Recommended default (Saved Question #1), mirroring **12.4 (Mary)** and **11.6/SplitPanel (Bob)**:

- **Present-all:** Sarah emits one `script_review` payload with `scripts: [<per-script entry>]` (Sarah already holds the full `self._generated_scripts`). The FE `SarahScriptReviewPanel` owns Prev/Next + the resolved set, exactly like `SplitPanel`/`MaryReviewPanel`.
- **Index/id-addressable approve/reject/skip:** because the user navigates freely, the actions carry `script_index` in `data` (default `_current_review_index` for back-compat). This pairs with a new `_reviewed_indices: set[int]` (Sarah's analog of Mary's `_reviewed_indices`), which becomes authoritative for the DONE gate (replacing the linear `_current_review_index >= len`).
- **Semantics preserved:** each script decided independently, no bulk-approve, mandatory review, DONE only after every index is reviewed. Only the transport + addressing change — the Epic-5 save/regenerate/advance/DONE *behavior* is unchanged; its *refinement* (feedback-into-prompt, approval metadata, Jack-eligibility) is **13.7**.

Alternative (Saved Q#1, if minimal backend change is preferred): keep Epic-5's one-at-a-time + backend `handle_navigate`; the FE renders the current `review_data` and round-trips a `navigate` message for Prev/Next. Trade-off: AC2's "review status visible for **each** item" is hard (the client only ever holds one item; it would have to accumulate as it navigates), and the UX diverges from Bob/Mary. The present-all default is recommended for consistency and to satisfy AC2 cleanly.
