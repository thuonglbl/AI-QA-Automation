# Decision Log

## 2026-04-03 — Product direction: AI QA Automation pipeline

- **Decision:** Build an AI-powered QA automation pipeline that converts natural-language Confluence test cases into executable Python Playwright scripts.
- **Rationale:** Reduce manual test-to-code translation effort, support non-coding QA testers, and preserve on-prem enterprise constraints.
- **Source:** PRD and product brief.
- **Status:** Current PRD decision

## 2026-04-03 — On-prem and enterprise-data constraint

- **Decision:** Keep test case content, generated scripts, browser sessions, and operational data within company-controlled infrastructure.
- **Rationale:** Target users operate in regulated Swiss enterprise contexts where cloud-only competitors are not viable.
- **Source:** PRD security and data sovereignty requirements.
- **Status:** Current PRD decision

## 2026-04-03 — Alice owns provider/model validation and rationale

- **Decision:** Alice dynamically validates the selected AI provider, discovers available models, blocks unverifiable model selections, and provides rationale for downstream agent model assignments.
- **Rationale:** Provider/model availability is dynamic and should not be hard-coded or centrally tuned by admins for MVP.
- **Source:** PRD FR15-FR15b.
- **Status:** Current PRD decision

## 2026-04-03 — Admin role scope

- **Decision:** Admin users manage users and projects, but are not responsible for switching LLM providers or centrally tuning prompt templates for users in MVP.
- **Rationale:** Provider selection and runtime model choices are user/workflow-scoped and dynamic.
- **Source:** PRD FR16-FR18.
- **Status:** Current PRD decision

## 2026-05-30 — Project-bound private conversation threads

- **Decision:** Introduce conversation threads. Each thread is created by one user, becomes bound to exactly one accessible project after Alice project selection, and cannot be rebound to another project.
- **Rationale:** Project-level singleton conversation state cannot support multiple concurrent workflows or user-private histories within the same project.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Threads are private; artifacts are project-shared

- **Decision:** Conversation threads remain private to the creating user, while generated artifacts are shared at project level for authorized project members.
- **Rationale:** Users on the same project need shared generated requirements, test cases, and test scripts without exposing each other's private conversation history.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Project-shared MinIO artifact tree

- **Decision:** Store generated artifacts in a project-scoped MinIO tree with required logical folders: `requirements`, `test_cases`, and `test_scripts`.
- **Rationale:** Multiple assigned users must be able to browse and continue from shared generated output regardless of artifact creator.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Encrypted per-user secrets

- **Decision:** Store AI provider keys and MCP keys as encrypted per-user PostgreSQL secrets, never in `.env`, `users.settings`, conversation history, artifacts, logs, WebSocket payloads, or generated files.
- **Rationale:** User-provided credentials belong to the user, require UI rotation, and must not be stored in plaintext JSON or leaked through runtime artifacts.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Non-secret provider and agent configuration only

- **Decision:** `ai_provider_config` and `ai_agents_config` may store selected provider, discovered model metadata, model assignments, runtime parameters, and rationale, but must not duplicate system-level environment settings or contain secrets.
- **Rationale:** Separate system configuration, user secrets, and non-secret model-selection metadata cleanly.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — `agent_runs` are thread-scoped

- **Decision:** Replace or rename project-scoped pipeline runs as `agent_runs`, scoped by `thread_id`; owning user and project are derived from the referenced conversation thread.
- **Rationale:** Execution state must not live on the project row because one project can have multiple private user threads.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Frontend collaborative workspace shell

- **Decision:** Replace the project-only frontend foundation with a collaborative workspace shell containing a collapsible sidebar, New Conversation, Conversation History, and a Project / Artifacts section for the active thread's bound project.
- **Rationale:** Users need to start new private threads, reopen prior threads, and browse shared project artifacts in one workspace.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Admin and standard-user workspace separation

- **Decision:** Standard users enter the collaborative workspace; admin users are routed only to the admin dashboard and do not enter the standard user workspace flow.
- **Rationale:** Admin responsibilities remain user/project management, while workflow execution and thread/artifact interaction are standard-user concerns.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved

## 2026-05-30 — Project artifact realtime sync

- **Decision:** Emit artifact change events for project artifact create, update, delete, or metadata changes; authorized users assigned to the changed project receive events and visible artifact trees refetch without resetting chat state.
- **Rationale:** Shared project artifacts must stay current across collaborators without disrupting the active conversation.
- **Source:** Approved sprint change proposal 2026-05-30.
- **Status:** Approved
